# Benchmark results

Boards are generated by `corpus/tools/gen-bench.mjs` (deterministic — regenerating reproduces the
files byte-for-byte); what each board measures is in its own `description` field. The `cpu` board
is a real-world working-CPU circuit (~4 k components, mixed D-FF/adder/gate types, clock-driven)
copied from `corpus/boards/cpu.json`.

## Protocol

```
taskset -c 0,2,4,6,8,10,12,14 ./target/release/sim bench corpus/bench/<board>.json \
    --ticks <N> --repeat 10
```

- **Machine**: i7-13700K (8 P-cores + 8 E-cores), Linux (Fedora), `intel_pstate/powersave`
  (boost active). Pinned to the 8 *physical* P-cores — one HT sibling each, no E-cores — so
  numbers don't depend on where the scheduler lands a worker.
- **Idle machine**: check `top` before running.
- The **best** ticks/s of 10 repeats is the recorded figure; `--ticks` sized per board so one
  repeat takes ~1–4 s.
- Bindings: `corpus/tools/bench-node.mjs` (Node addon, `runAsync`, release build),
  `corpus/tools/bench-wasm.mjs` (wasm blocking `run()`, nodejs-target release build), and
  `corpus/tools/bench-cpp-node.mjs` (the **old C++ engine**, `@logigator/logigator-simulation`
  1.2.1 — the rewrite's reference point).
- Acceptance per phase: the stated target boards improve by the stated hypothesis, and no board
  regresses by more than ~2% (the noise floor; re-run before concluding a real regression).


### Node addon

Build (once per code change; must be a release build — debug measures the wrong thing):

```
cd crates/sim-node && npx napi build --platform --release
```

Run (same P-core pinning as the CLI):

```
taskset -c 0,2,4,6,8,10,12,14 node corpus/tools/bench-node.mjs corpus/bench/<board>.json \
    --ticks <N> --repeat 10
```

Each repeat constructs a fresh `Simulation` (power-on init) and times one bounded `runAsync` call
on a dedicated worker thread. `--threads` defaults to 1; omit it for like-for-like comparisons
with the CLI ST numbers.

### WASM (nodejs target)

Build (once per code change; simd128 required for a representative number):

```
RUSTFLAGS="-C target-feature=+simd128" wasm-pack build crates/sim-wasm --release \
    --target nodejs --out-dir pkg-bench-node -- --no-default-features --features serde
```

Run:

```
taskset -c 0,2,4,6,8,10,12,14 node corpus/tools/bench-wasm.mjs corpus/bench/<board>.json \
    --ticks <N> --repeat 10
```

Each repeat constructs a fresh `Simulation` and times the blocking `run()` call. Note: `run()`
invokes `performance.now()` on every tick (the wasm timeout path has no amortization equivalent to
P1), so idle-board numbers are structurally lower than the CLI and do not reflect the same
optimization headroom. Node-hosted wasm is a stand-in for the browser (same V8 JIT); re-measure
in a real browser if a decision ever hinges on it.

## Results

### P0 — Baseline (commit 906d84e, rustc 1.96.0)

Establish a measurement substrate before touching any code: synthetic boards spanning the perf
regimes, a results file, and a `perf` profile pinning the known hotspots as a reference for
later phases.

The multi-threaded (MT) driver was benchmarked here for completeness; it was a net loss or wash
on every board and was removed in P1.5. Node addon numbers at P0 are already fast on idle boards
because `runAsync` batches ticks on a dedicated thread and never called `clock_gettime` per tick;
the CLI P1 fix closes that gap. WASM numbers are structurally lower than CLI because `run()` reads
`performance.now()` on every tick for its JS-side timeout check.

### P1 — Amortize per-tick clock reads (commit 9f8c0d0, rustc 1.96.0)

`run_single` called `clock_gettime` (~15–25 ns) on every tick via `update_speed`. On idle
boards a tick is itself only tens of ns, so the clock read dominated. Fix: check the clock only
every ~1024 ticks (matching how the Node worker already batched), accepting up to 1023 ticks of
timeout overshoot.

CLI and WASM gain sharply on idle boards (the clock overhead ran in Rust on both paths); active
boards are flat (tick work dominates). Node was already amortised and sees only noise on idle
boards; active boards track CLI.

### P1.5 — Remove the adaptive parallel driver (commit aae8724, rustc 1.96.0)

P0 showed MT was a net loss where it engaged (`fanout` −17–32%, `large_active` +10% at 8
threads at ~20% parallel efficiency). The MT8 profile attributed the loss to atomic
`set_output` traffic and the serial merge — structural costs that couldn't be tuned away — while
the allocation churn P4 had targeted was only ~2% of MT time. Rather than tune an unfixable
driver, deleted it entirely.

No regression; small gains on boards that carried the threads-enabled bookkeeping tax
(`medium_active`, `medium_idle`, `correlated`). Node and WASM track CLI; the driver removal had
no surface-level effect on the binding paths.

### P2 — Pre-bucket link consumers by type (commit 927b805, rustc 1.96.0)

The read phase's consumer enqueue loop loaded `comp_ty_index[c]` per consumer (a random-access
gather) then pushed to a per-type queue one element at a time. Fix: at compile time, sort each
link's consumers by type and record the runs as `(type_index, len)` groups. The read phase then
bulk-copies each group with `extend_from_slice` — no per-element type lookup, no per-element
capacity check, and the consumer slice streams sequentially.

Multi-consumer boards win: `fanout` (each ring link drives 2500 same-type sinks) and `correlated`
(mixed multi-input components with correlated input flips) gain across all three surfaces. The
`cpu` board (real-world mixed circuit) also gains noticeably. All `*_active` pure NOT rings are
fan-out-1, so the grouping is a no-op for them.

### P3 — Compute-queue dedup (NOT LANDED — implemented, measured, reverted; engine = P2)

Multi-input components (adders, FFs with data and clock) can be enqueued for recompute multiple
times in one tick when several inputs flip together. Deduplicating the queue — via a
`compute_queued` bitset or per-component tick stamps — would avoid redundant kernel calls.

Both a `compute_queued` bitset and a pay-as-you-go tick-stamp variant were tried. Both
configurations regressed every board, including `correlated` (the only board with duplicates at
15.1% rate): duplicate computes hit idempotent kernels (~25 cycles each), while the dedup test
runs on every enqueue attempt at ~3–4 cycles — near breakeven per avoided duplicate, negative
once counting the branch overhead, stamp-array working-set cost, and codegen perturbation.
Revert `ba75800`; recoverable from `2c811d3` / `a0f0001`.

### P4 — Kill parallel-path allocation churn (DROPPED — superseded by P1.5)

The per-`run` rayon pool rebuild and the per-tick frontier `Vec` reallocation in the MT path
were hypothesized as the main sources of MT overhead. The fix was persistent per-thread scratch
buffers and a kept-alive pool.

The P0 MT8 profile of `large_active` falsified the premise: `memmove` (the realloc/copy churn
this phase targeted) was ~2% of MT CPU time. The structural costs — atomic `set_output` traffic
(~36%) and the serial merge + orchestration (~13.5%) — cannot be addressed by scratch reuse.
P1.5 removed the driver entirely; P4 was dropped with it. Recoverable only as a concept; no
code was written.

### P5 — Component locality renumbering (commit 7532ee9, then reverted in 7239a7a)

Component ids follow submission order, scattering a given type's data (CSR slices, scratch
slots) across memory. Assigning contiguous internal ids per type at compile time — behind a
`pub2int`/`int2pub` translation — would make a type's dirty-queue accesses stride sequentially
through CSR and scratch rather than scatter across them.

Type-bucketed renumbering was perf-neutral on all boards: the `*_active` rings are type-
homogeneous (bucketing is a no-op permutation), and the mixed-type boards (`correlated` −0.3%,
`cpu` +0.3%) already have working sets that fit L2. A queue-sort follow-on regressed active
boards 38–52% and was reverted separately. The renumbering itself was reverted after P6 was also
reverted, removing its only structural consumer. Recoverable from `7532ee9`.

### P6 — Word-level read phase + single-driver specialization (NOT LANDED — implemented, measured, reverted; engine = P2)

The read phase loaded `driver_count[l]` per scheduled link to detect real state changes. The
proposed fix: at write time, record the link's resolved next value in a `next_link_state` bitset
directly (single-driver links bypass `driver_count` entirely; multi-driver links maintain it but
only on 0↔1 crossings) and schedule the link's 64-bit *word* into a dirty-word list. The read
phase then XORs word pairs to find changed bits and walks them with `trailing_zeros` — no
per-link count loads, no duplicate link pushes.

All boards regressed 7–17% across two implementations (`110a96c`, `d802605`). The read phase
did remove `driver_count` gather traffic as intended, but consecutive `set_output` calls on
dense boards serialize on a store-to-load-forwarding chain through the shared `next_link_state`
word (each RMW waits on the previous flip's store), where the per-link path hits independent
addresses and pipelines freely. Revert `6819c45`; recoverable from `d802605`. The pending-mask
boundary discipline any future word-granular attempt must reuse is documented in `d802605`'s
commit message and the behavior tests added for this phase stay as permanent pins.

### Cleanup — Remove dead AVX2 gather path (engine = P2)

`simd.rs` contained a `vpgatherdd`-based gather (`gather_word_avx2`) that fired only for gates
with ≥ 256 inputs. Such gates do not occur in real circuits, so the branch was dead code.
The runtime `is_x86_feature_detected!("avx2")` check ran on every AND/OR/XOR gate evaluation
and always branched false. Removed the AVX2 function, the `WIDE_FANIN` constant, the dispatch
macro, and the two AVX2-specific tests. The scalar gather-reduce path is unchanged.
